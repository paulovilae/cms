## ⚠️ Potential Pitfalls & Mitigation Strategies

### 🚨 Critical Risks

#### 1. Information Overload
- **Risk**: Too much information in too many places
- **Early Warning Signs**: 
  - info.md files exceeding 1000 lines
  - Agents reporting difficulty finding information
  - Duplicate information across multiple locations
- **Mitigation Strategies**:
  - Implement knowledge curation workflows
  - Automated archival of outdated information
  - Regular knowledge audits and cleanup
- **Emergency Response**: Temporary freeze on new info.md creation until cleanup

#### 2. Knowledge Fragmentation
- **Risk**: Important insights buried in obscure locations
- **Early Warning Signs**:
  - Repeated questions about previously documented solutions
  - Agents recreating existing solutions
  - Inconsistent information across locations
- **Mitigation Strategies**:
  - Automated cross-referencing systems
  - Semantic search capabilities
  - Regular knowledge consolidation reviews
- **Emergency Response**: Knowledge archaeology project to recover buried insights

#### 3. Security Boundary Erosion
- **Risk**: Gradual expansion of permissions beyond intended scope
- **Early Warning Signs**:
  - Agents requesting access outside their domain
  - Unauthorized file modifications
  - Permission escalation requests
- **Mitigation Strategies**:
  - Automated permission monitoring
  - Regular security audits
  - Principle of least privilege enforcement
- **Emergency Response**: Immediate permission rollback and security review

#### 4. Encryption Key Management Failure
- **Risk**: Lost access to encrypted knowledge
- **Early Warning Signs**:
  - Key rotation failures
  - Agent authentication issues
  - Master key access problems
- **Mitigation Strategies**:
  - Multiple key backup locations
  - Automated key rotation procedures
  - Emergency recovery protocols
- **Emergency Response**: Activate emergency recovery keys and rebuild encryption

#### 5. Knowledge Quality Degradation
- **Risk**: Inaccurate or misleading information proliferation
- **Early Warning Signs**:
  - Conflicting information in different locations
  - Outdated solutions being referenced
  - Agent confusion about best practices
- **Mitigation Strategies**:
  - Peer review systems
  - Version control for knowledge
  - Quality scoring mechanisms
- **Emergency Response**: Knowledge quarantine and expert review

## 🛠️ Debugging & Troubleshooting

### Common Issues & Solutions

#### Issue: "Invalid mode configuration" / "Invalid input"
- **Symptoms**: Agent fails to load with YAML validation error
- **Root Cause**: Incorrect permission structure format
- **Solution**: 
  - For unrestricted agents: Use `- edit` (single dash)
  - For restricted agents: Use `- - edit` with nested restrictions
  - Verify `source: project` is present and last
- **Prevention**: Always verify permission structure matches access level needed

#### Issue: "Map keys must be unique"
- **Symptoms**: YAML parser error about duplicate keys
- **Root Cause**: Multiple `source: project` fields in same file
- **Solution**: Ensure exactly one `source: project` field at the end of file
- **Prevention**: Use YAML validator before deployment

#### Issue: "All mapping items must start at the same column"
- **Symptoms**: YAML indentation error
- **Root Cause**: Inconsistent indentation in YAML structure
- **Solution**: Use consistent 2-space indentation, align all top-level keys
- **Prevention**: Use automated YAML formatting tools

### Systematic Verification Checklist
When an agent fails to load:
1. **Check YAML syntax**: Use YAML validator
2. **Verify permission structure**: Match pattern to access level needed
3. **Confirm required fields**: Ensure `source: project` is present and last
4. **Check indentation**: Use consistent 2-space indentation
5. **Compare with working profile**: Use similar agent as template
6. **Apply systematic verification**: Check if other agents need same fix
